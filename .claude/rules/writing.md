# 教程文档写作规范

## 结构要求

1. **小章节原则**：每份文档尽可能小，拆分文件分包，一次只看一小部分
2. **文档命名规范**：`序号字母-话题前缀-侧重内容.md`。同一话题共用一个序号，a/b/c 字母细分，话题词作为前缀。正文在先（a, b, c...），补充在后（后续字母）
3. **前置知识按需补充**：前置理论仅针对当前文档正文涉及的内容，正文未涉及的概念不在当前补充，避免增加不必要负担
4. **知识体系推进**：先基础后复杂，按顺序讲解，不跳跃
5. **先概念后实现，理论必回归源码**：先说"是什么"，再说"怎么实现"。先讲数据结构，再讲方法逻辑。理论后必须引用具体源码（文件路径+行号）
6. **顶层优先**：先讲顶层抽象组成部分，最后才讲具体实现
7. **多用具体实例**：抽象概念必须配具体例子（如"student 表有 id INT, name STRING, age INT"）
8. **文档单一职责**：每个文档只包含当前主题，无关内容单独新建文档
9. **追问多的概念单独成文**：连续追问多次的概念应提取为独立文档，原文档中引用
10. **结尾导航链接**：每个文档末尾必须有 `上一节：[文件名](./文件名) | 下一节：[文件名](./文件名)`，用文件名作为显示文本，不用中文标题

## 内容要求

11. **短段落原则**：尽量一句一段。一个句号之后另起一段，避免大段文字堆在一起。短段落让读者更容易阅读，也帮作者理清"每句话在说什么"。

12. **零基础友好**：假设读者是刚入门的小白，补充必要前置概念
12. **上下文串联**：正文讲完后简要串联相关模块的上下文关系（对上谁使用、对下依赖谁）
13. **明确输入输出**：每层给出输入类型和输出类型（如"输入：SQL 文本 → 输出：AST"）
14. **层间对接要具体**：说清楚通过什么接口/方法调用、传递什么类型的数据
15. **对比分析要讲优缺和问题**：每个方案分别给出优点、缺点、潜在问题、适用场景
16. **源码优先 + 概念可视化**：实现细节用源码，抽象概念用 mermaid 或文本框图
17. **不适合 mermaid 的用文本块**：如项目结构不适合画图，用文字版文件树
18. **缩写全称**：所有缩写给出全称（如 RM = Record Manager）
19. **原文优先**：只有源码中的名称才给英文原文再中文解释，普通文字直接用中文
20. **标题即主题**：标题只含主题词，不含破折号、括号、英文翻译

## 质量要求

21. **答疑紧接正文，补充放在正文后**：答疑和补充说明跟在对应正文后面。只有前置知识放正文前面
22. **随时能出版**：风格统一，排版统一，不能是草稿
23. **引用一致性**：文件索引和引用要与实际对应，发现对不上及时调整
24. **自审完整性**：发现没有上下文突然出现的内容，及时补充
25. **维护文档聚焦最近查看**：用户说"维护文档"时只检查当前打开和直接关联的文档
26. **文件名匹配内容**：维护时检查文件名是否反映文档覆盖范围，不匹配一并调整并更新引用
27. **先验证理解再调整**：用户要求调整内容时先判断其理解是否正确，防止盲从错误
28. **保留生僻字拼音**：文档中的拼音标注不能删除，是用户有意添加的

## 格式规范

29. **引用块格式工整**：`>` 引用块（答疑备注、补充说明）里面的内容格式也要工整——该用列表就用 `-` 列表，该用表格就用表格，不能一大段文字堆在一起。多维度对比用表格，枚举要点用列表。
