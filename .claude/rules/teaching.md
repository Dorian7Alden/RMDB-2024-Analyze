# 教学方法论

## 核心学习方法

**以框架待实现部分为学习目标**：`db2026-x/` 是空的题目框架，`src/` 是完整参考实现。学习重点是框架中标注 TODO 或方法体为空的待实现部分，对比两边的差异来讲解"要做什么"和"怎么做"。框架已完整实现的内容只做简要介绍。

**框架代码不要过度解释**：`db2026-x/` 是比赛用的题目框架，索引、事务等模块都只是给出了骨架，需要参赛选手自行设计、自行考量实现方案。因此文档中对框架代码只描述"它大概有什么、定义了哪些结构"，**不深入解释框架的实现逻辑**。重点放在参考实现 `src/` 的设计思路上，让学习者看懂"为什么要这么做"，自己去写框架里的代码。

## 讲解顺序规则

1. **先概念后实现**：先讲"是什么东西"，再讲"怎么实现的"。讲解每层时先讲数据结构（涉及哪些类、结构体、成员变量），再讲实现（方法逻辑、算法流程）

2. **每层必加交互文档**：每个模块/层讲解完各个组件之后，必须有一篇文档专门梳理**本层内部各组件之间的交互机制**——谁调用谁、谁依赖谁、数据怎么流转。内容包括：组件分工表、调用关系图、典型工作流的调用链、数据流向图、依赖层次。让读者看完单个组件后，能用这篇"串起来"，形成整体认知。如记录层的 `08-record-interaction.md`。

3. **每层必加 API 速查文档**：讲解完一个层之后，补充一个"API 速查"文档，列出该层每个类的核心接口（方法签名 + 一句话描述），方便读者快速回顾和加深印象。放在总结文档之前。如记录层的 `11-record-api-reference.md`。

4. **讲解新的层时联系已学层**：开始讲解新的一层时，主动联系前面学过的层。对比相似之处（如 `.db` 和 `.idx` 都是定长页面文件，都按 Page 加载，都有文件头和页头），分析差异之处（如 Bitmap vs Keys数组、空闲链表 vs B+树结构）。让读者感觉到知识是连贯的，不是在学孤立的知识点。

5. **概念讲"是什么"再"为什么"**：引入新概念必须先解释概念本身，再讲应用场景和原因。读者连概念都不知道时，理解不了为什么要用。讲完"是什么"和"为什么"之后，**必须给出这个概念的实际上使用场景**——在哪个文件、哪个方法、什么流程中用到的。不能只讲概念本身而不讲"在哪里用"。

6. **结合应用场景，不能纯讲概念**：讲解任何数据结构、函数、类时，不能只介绍"它是什么、怎么实现的"，必须配套说明"在哪里被调用、被谁使用"。比如讲了 `Iid` 不仅要说它和 `Rid` 的区别，还要说它被 `IxScan` 用于范围扫描的起止边界；讲了 `IxFileHdr` 不仅要说有哪些字段，还要说 `first_leaf_` 被 `IxScan` 用作起始位置。

7. **源码优先**：引入新概念必须先展示相关源码，再解释代码含义。源码是第一手材料，讲解是辅助理解源码的

8. **理论回归源码**：理论讲完后必须引用具体源码（文件路径+行号）加深印象，最终目标是看懂源码、手写源码

## 框架对比方法

9. **对比差异要分析原因**：框架 vs 参考实现的差异不能只列清单，必须讲清楚"为什么这么改"

10. **对比分析要讲优缺点**：多种策略对比时各自给出优点、缺点及潜在问题，每个方案说清楚什么场景适用、什么场景会出问题，不偏向某一方

## 教学行为规则

11. **先验证理解再调整**：用户要求调整内容时，先判断其理解是否正确。理解有误则先解释纠正，确认后再调整；防止盲从错误理解把文档改错

12. **识别并持久化规则**：用户每次给出的调整要求、反馈意见中，凡是可以复用于后续工作的，都应自动识别为"规则"写入配置

13. **规则优先于文档修改**：用户要求调整/记住规则时，先更新规则文件再改文档
