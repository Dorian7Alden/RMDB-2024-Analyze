# 事务与并发总结

## 本章覆盖范围

**含义**：第 6 章讲的是数据库如何保证多个事务同时执行不会互相破坏数据。

**范围**：本章覆盖 Transaction、TransactionManager、LockManager、间隙锁和各个模块之间的调用链。

## 核心数据流

**含义**：事务并发控制的本质是对写操作记录做反向执行，对锁做兼容检查。

| 组件 | 角色 | 核心数据结构 |
|------|------|-------------|
| Transaction | 事务运行时状态容器 | write_set_、lock_set_、state_ |
| TransactionManager | 事务生命周期管理入口 | txn_map、next_txn_id_、next_timestamp_ |
| LockManager | 并发读写保护 | lock_table_、gap_lock_table_、latch_ |
| WriteRecord | 撤销恢复的最小单元 | wtype_、tab_name_、rid_、record_ |
| LockDataId | 锁资源唯一标识 | fd_、rid_、index_meta_、gap_、type_ |
| LockRequestQueue | 单资源上的锁等待队列 | request_queue_、cv_、group_lock_mode_ |

## 和前面五章的关系

**含义**：事务层是前五章的安全控制层。

| 已学章节 | 本章如何使用 |
|----------|--------------|
| 存储层 | 缓冲池本身有 RLatch/WLatch 的 Page 级锁，事务层的锁保护更细粒度的记录和范围 |
| 记录层 | 事务的回滚需要反向操作 RmFileHandle 的 insert、delete 和 update |
| 索引层（IX 模块） | 回滚时需要反向调用索引层的 delete_entry 和 insert_entry，间隙锁直接绑定索引 |
| 系统层 | TransactionManager 持有 SmManager 指针，通过它获取表元数据和文件句柄 |
| 查询处理层 | 执行层的每个算子都可能调用 LockManager 加锁，并在修改后追加写集 |

## 框架学习重点

**含义**：`db2026-x/` 中事务层的待实现内容是 LockManager 的全部方法和 TransactionManager 的 commit 与 abort。

| 优先级 | 项目 | 原因 |
|--------|------|------|
| 1 | 表级锁 IS、IX、S、X | 最简单的锁粒度，理解加锁和冲突检查的基本框架 |
| 2 | lock_shared_on_record | 记录级读锁，理解 wait-die 死锁预防 |
| 3 | lock_exclusive_on_record | 记录级写锁，理解锁升级 |
| 4 | unlock | 锁释放，理解从 GROWING 到 SHRINKING 的切换 |
| 5 | commit | 写集中已有的记录不再需要回滚了，清理即可 |
| 6 | abort | 最复杂的操作：逆序遍历写集、反向修改、索引维护 |
| 7 | 间隙锁相关 | 可串行化隔离中的进阶内容，阻挡幻读 |

**建议**：先跑通一个单线程 INSERT + COMMIT，确认 commit 路径正常；再跑两个并发事务的 SELECT，确认锁不掐死对方；最后试 UPDATE 回滚。

## 关键设计思路

### 两阶段封锁

**含义**：两阶段封锁强制事务在加锁期间不能释放锁，在释放锁后不能重新加锁。

**作用**：它防止了并发冲突中出现不可预测的执行顺序，是实现可串行化的基础。

**示例**：如果允许事务先释放表锁再申请表级排他锁，两个事务就可能互相插队，导致调度不可串行化。

### wait-die 死锁预防

**含义**：根据事务的时间顺序决定冲突时的取舍——老事务等待年轻事务，年轻事务遇到老事务直接中止。

**作用**：它避免了环形等待的发生，不需要死锁检测就能预防死锁。

**示例**：T1 比 T2 老，T1 想拿 T2 持有的锁时等待，T2 想拿 T1 持有的锁时直接中止。

### 多粒度锁

**含义**：多粒度锁在同一张表上同时有表级意向锁和记录级锁。

**作用**：它兼顾了整表操作的效率和单记录操作的并发性。

**示例**：T1 在表上加 IS 锁并在记录上加 S 锁读数据；T2 想锁整张表做全表扫描时，只需检查表上的锁——看到 IS 锁就知道和 X 冲突，不用扫表中每一行。

### 写集回滚

**含义**：事务的原子性通过逆序执行写集保证。

**作用**：回滚时不需要分析逻辑上的依赖关系，只需按写入的逆序操作。

**示例**：如果事务先 INSERT 后 UPDATE，回滚时先恢复 UPDATE 再撤销 INSERT，保证中间状态一致。

### 间隙锁与幻读

**含义**：间隙锁是对索引范围加锁，阻止其他事务在范围中间插入或删除新键。

**作用**：它是可串行化隔离级别下防幻读的手段。

**示例**：T1 扫描 `age > 18` 时对 `[18, INF)` 加 S 间隙锁，T2 在此期间不能插入任何 `age > 18` 的记录。

## 锁分类总结

| 维度 | 表锁 | 记录锁 | 间隙锁 |
|------|------|--------|--------|
| 级别 | 事务级 | 事务级 | 事务级 |
| 范围 | 整张表 | 单条记录 | 索引范围 |
| 类型 | IS、IX、S、SIX、X | S、X | S、X |
| 生命周期 | 算子构造时申请，事务结束时释放 | 访问前申请，事务结束时释放 | 索引扫描前申请，事务结束时释放 |

## 常见误区

| 误区 | 正确认识 |
|------|----------|
| 事务锁等于存储层页锁 | 事务锁保护的是逻辑数据，页锁保护的是物理页面读写 |
| 可串行化等于没有并发 | 记录级锁和间隙锁可以让不冲突的事务真正并发执行 |
| SHRINKING 阶段是释放锁 | 两阶段锁要求的是“先全部加锁、再全部释放”，中间不能交叉 |
| commit 需要真正写入数据 | 数据在执行层执行时已经写入，commit 只需要确认和清理 |
| 间隙锁越多越好 | 间隙锁是昂贵的——它阻止了非冲突事务的插入，只应在可串行化隔离下使用 |
