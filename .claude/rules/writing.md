# 教程文档写作规范

## 结构要求

1. **小章节原则**：每份文档尽可能小，拆分文件分包，一次只看一小部分
2. **文档命名规范**：`序号字母-话题前缀-侧重内容.md`。同一话题共用一个序号，a/b/c 字母细分，话题词作为前缀。正文在先（a, b, c...），补充在后（后续字母）
3. **前置知识按需补充**：前置理论仅针对当前文档正文涉及的内容，正文未涉及的概念不在当前补充，避免增加不必要负担
4. **知识体系推进**：先基础后复杂，按顺序讲解，不跳跃
5. **先概念后实现，理论必回归源码**：先说"是什么"，再说"怎么实现"。先讲数据结构，再讲方法逻辑。理论后必须引用具体源码（文件路径+行号）。引用方式分两种，**二选一，不可重复**：
   - **有代码块**：源码位置在代码块内部第一行用注释标注（如 `// src/index/ix_index_handle.cpp:100`）。代码块外部**不得**再用 `**源码**：` 标签或文字重复标注——内部注释已是唯一且充分的来源说明
   - **无代码块**：在正文中用 `**源码**：` 标签或内联引用（如 `src/record/rm_defs.h:24-31`）给出文件路径+行号，让读者自行跳转查看
6. **先抽象再具体，最后串联**：讲解任何内容必须遵循三层递进结构，不可跳过：

   - **第一层：抽象概览**。先说"这整件事是什么"——用一个 mermaid 流程图或一段概述给出全局视角。读者看完这一层应该知道整体轮廓：输入什么、输出什么、分几个阶段、各阶段什么关系
   - **第二层：逐块拆解**。把概览中的每个阶段拿出来，逐一深入讲解。每块的讲解也遵循同样的模式——先说这块做什么，再展开具体实现
   - **第三层：串联回顾**。所有块讲完后，回到抽象概览的视角，把各块之间的调用关系、数据流向串起来，让读者形成完整认知

   禁止的做法：一上来就直接讲某个具体方法或某段代码。读者不知道这个方法在整个流程中处于什么位置、解决什么问题，看了也白看。
7. **多用具体实例**：抽象概念必须配具体例子（如"student 表有 id INT, name STRING, age INT"）
8. **文档单一职责**：每个文档只包含当前主题，无关内容单独新建文档
9. **追问多的概念单独成文**：连续追问多次的概念应提取为独立文档，原文档中引用
10. **结尾导航链接**：每个文档末尾必须有 `上一节：[文件名](./文件名) | 下一节：[文件名](./文件名)`，用文件名作为显示文本，不用中文标题

## 内容要求

11. **短段落原则**：尽量一句一段。一个句号之后另起一段，避免大段文字堆在一起。短段落让读者更容易阅读，也帮作者理清"每句话在说什么"。

12. **零基础友好**：假设读者是刚入门的小白，补充必要前置概念
13. **上下文串联**：正文讲完后简要串联相关模块的上下文关系（对上谁使用、对下依赖谁）
14. **明确输入输出**：每层给出输入类型和输出类型（如"输入：SQL 文本 → 输出：AST"）
15. **层间对接要具体**：说清楚通过什么接口/方法调用、传递什么类型的数据
16. **对比分析要讲优缺和问题**：每个方案分别给出优点、缺点、潜在问题、适用场景
17. **源码优先 + 概念可视化**：实现细节用源码，抽象概念用 mermaid 或文本框图
18. **不适合 mermaid 的用文本块**：如项目结构不适合画图，用文字版文件树
19. **缩写全称**：所有缩写给出全称（如 RM = Record Manager）
20. **原文优先**：只有源码中的名称才给英文原文再中文解释，普通文字直接用中文
21. **标题即主题**：标题只含主题词，不含破折号、括号、英文翻译

## 质量要求

22. **答疑紧接正文，补充放在正文后**：答疑和补充说明跟在对应正文后面。只有前置知识放正文前面
23. **随时能出版**：风格统一，排版统一，不能是草稿
24. **引用一致性**：文件索引和引用要与实际对应，发现对不上及时调整
25. **自审完整性**：发现没有上下文突然出现的内容，及时补充
26. **维护文档聚焦最近查看**：用户说"维护文档"时只检查当前打开和直接关联的文档
27. **文件名匹配内容**：维护时检查文件名是否反映文档覆盖范围，不匹配一并调整并更新引用
28. **先验证理解再调整**：用户要求调整内容时先判断其理解是否正确，防止盲从错误
29. **保留生僻字拼音**：文档中的拼音标注不能删除，是用户有意添加的

## 格式规范

30. **引用块格式工整**：`>` 引用块（答疑备注、补充说明）里面的内容格式也要工整——该用列表就用 `-` 列表，该用表格就用表格，不能一大段文字堆在一起。多维度对比用表格，枚举要点用列表。

31. **明确区分概念，防止混淆**：讲解 B+ 树、Map 等涉及 key-value 的模块时，严格区分以下概念：
   - **键（key）**：索引字段的值，如 `age=20` 中的 `20`
   - **键值对（key-value pair）**：key 和 value 的组合
   - **避免单独使用"键值"**：这个词容易混淆——可能被理解为"key 的值"或"键值对"。如果指 key，直接用"键"；如果指 pair，用"键值对"
   - **避免"键值"作形容词**：如"键值数组"应是"键数组"

32. **指示性标签引导阅读**：每段解释性内容必须先给出指示性标签，再展开描述。读者不应自行猜测"这段在讲什么"。

    常用标签及适用场景：

    | 标签 | 适用场景 |
    |------|---------|
    | **含义** | 解释概念/术语的定义 |
    | **作用** / **用途** | 说明方法/数据结构的用处 |
    | **场景** | 在什么流程中被调用、被谁使用 |
    | **实现** | 用什么算法/技术实现 |
    | **源码** | 紧接着给出文件路径 + 行号 |
    | **输入** / **输出** | 参数和返回值的含义 |
    | **示例** | 给出具体例子 |
    | **对比** | 和相似概念的区别 |

    反面例子（读者需自行猜测内容性质）：
    > `find_leaf_page` 从根节点出发，逐层搜索直到叶节点。

    正面例子（标签明确指示内容性质）：
    > **含义**：`find_leaf_page` 是 B+ 树的逐层下钻操作——从根节点出发，每一层用 `internal_lookup` 确定走哪个孩子，直到到达叶节点。
    > **场景**：被 `get_value`、`insert_entry`、`delete_entry` 调用，是所有 B+ 树操作的公共前缀。
    > **源码**：`src/index/ix_index_handle.cpp:270`（参考实现）。
