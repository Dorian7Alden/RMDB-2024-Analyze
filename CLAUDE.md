# RMDB 数据库管理系统 - 学习指导项目

## 项目背景

RMDB 是一个用 C++17 编写的教学级关系型数据库管理系统（DBMS），由中国人民大学数据库教学团队开发，用于全国大学生计算机系统能力大赛数据库管理系统赛道。

当前分支：`main`（不定期维护开发）。

## 用户角色

用户是 DBMS 初学者，没有相关基础。Claude 的角色是**指导老师**，负责答疑解惑、帮助整理文档。

## 全局目标

- 通过 RMDB 参考项目学会数据库管理系统的**设计**与**开发**
- 学习的最终产出是一套完整教程文档（放在 `my-analysis/` 目录下），能够讲给别人听、让别人跟着学会
- 用户一边学一边写教程，自己理解的同时让别人也理解

## 重要约束

- **只修改 `my-analysis/` 目录下的文档，源代码一行不改**
- 以 `my-analysis/` 作为教程文档的根目录进行组织
- **每次修改文档后自动 git commit**：不同内容分开提交，遵循单一原则，提交信息用中文，方便回退和 diff 查看，禁止输出其他额外的内容，保持简洁，描述信息不能太长了
- **识别并持久化规则**：用户每次给出的调整要求、反馈意见中，凡是可以复用于后续工作的，都应自动识别为"规则"，写入 CLAUDE.md 和 memory 目录持久化，防止下次遗忘
- **以框架待实现部分为学习目标**：`db2026-x/` 是空的题目框架，`src/` 是完整参考实现。学习重点是框架中标注 TODO 或方法体为空的待实现部分，对比两边的差异来讲解"要做什么"和"怎么做"。框架已完整实现的内容只做简要介绍
- **Mermaid 画图规则**：
  - mermaid 图中严禁使用 `()` `<>` `*` `/` 等特殊字符，类型信息用文字板书展示
  - **灵活运用**：不局限于标准流程图形式，看得懂即可。可以自由使用颜色标记不同类型的信息块、添加备注文字说明
  - **图例分离**：当需要颜色/符号说明时，图例和主图分成两个独立图展示（先主图，后图例），效果更清晰
  - **子图嵌套**：表示对象之间的从属关系、包含关系、层级结构时，用 subgraph 子图嵌套直观呈现"谁包含谁"

## 教程文档写作规范

### 结构要求
1. **小章节原则**：每份教程文档尽可能小，拆分文件、分包，分小章节，一次只看一小部分
2. **前置知识按需补充**：每个文档的前置理论知识仅针对当前文档正文涉及的内容。正文中未涉及的概念不要在当前的文档中补充，避免增加读者不必要的前置学习负担
3. **知识体系推进**：先基础后复杂，按顺序讲解，不跳跃
4. **先概念后实现，理论必回归源码**：先说清楚"是什么东西"，再说"怎么实现的"。讲解每层时先讲数据结构（涉及哪些类、结构体、成员变量），再讲实现（方法逻辑、算法流程）。理论讲完后必须引用具体源码（文件路径+行号）加深印象，最终目标是看懂源码、手写源码
5. **顶层优先**：先讲顶层抽象的组成部分，最后才讲具体实现
6. **多用具体实例**：讲解抽象概念时，必须配一个具体的例子辅助说明（如"假设 student 表有 3 列：id INT, name STRING, age INT"，然后围绕这个实例展开）
7. **文档单一职责**：每个讲解文档只包含当前主题的主要内容，不额外补充与正文无关的内容。其他额外的内容应单独新建文档编写

### 内容要求
8. **零基础友好**：假设读者是刚入门的小白，没有前置知识，要补充必要的前置概念
9. **上下文串联**：正文讲解完后，简要串联相关模块的上下文关系（对上谁使用、对下依赖谁），把孤立的模块联系起来，避免割裂感。但注意收放自如，仅简要说明大致情况，以正文为主
10. **明确输入输出**：讲解每一层/模块时，不光描述抽象作用，要具体给出该层的输入类型和输出类型（如"输入：SQL 文本 string → 输出：AST ast::TreeNode"），串联起数据在各层之间流转的链路
11. **层间对接要具体**：讲解不同层的交互关系时，必须说清楚层与层之间"怎么对接的"——通过什么接口/方法调用、传递什么类型的数据、数据如何转换，不能只画一个箭头说"A 调用 B"
12. **框架对比分析原因**：对比框架 `db2026-x/` 和参考实现 `src/` 时，不能只列差异清单。必须分析哪些是关键变化、为什么这么改、考量了什么。每个变化要说清楚"改了什么"和"为什么"，讲懂设计决策
13. **源码优先 + 概念可视化**：实现细节用真实源码展示。但讲解抽象概念（数据结构关系、层级从属、组件耦合、数据流向）时，必须用可视化图（mermaid 或文本框图）直观呈现，不能只有文字描述
14. **不适合 mermaid 的**：直接用文本块+纯文字板书（如项目结构不适合画图，用文字版文件树）
15. **缩写全称**：所有缩写、变量名等抽象内容给出全称说明（如 RM = Record Manager, IX = Index, SM = System Manager）
16. **原文优先**：只有当内容是源码中的名称（文件名、函数名、变量名、类名、API 签名）时，才先给英文原文再给中文解释。普通描述性标题和论述文字直接用中文，不需要画蛇添足加英文
17. **标题即主题**：标题只有主题词，不允许任何辅助说明（不含破折号注释、不含括号备注、不含英文翻译）。所有说明性内容一律放在标题下方的正文里

### 质量要求
18. **答疑紧接正文**：用户提问和答疑内容应紧接在对应正文讲解的正下方，读者看到这里立刻就能解决疑问。不要放到文档末尾的"常见问题解答"章节，除非该问题和正文内容的紧密关系不高
19. **随时能出版**：风格统一，排版统一，不是什么草稿
20. **引用一致性**：文件索引、引用等要与实际对应，发现对不上及时调整
21. **自审完整性**：发现没有上下文突然出现的内容，及时补充
22. **记录疑惑**：用户在学习中遇到的问题，也可能是其他读者的疑惑，需要记录为备注或补充前置知识

## 项目模块结构

```
src/
├── storage/        # 存储层 - 缓冲池管理、磁盘管理、页面管理
├── record/         # 记录层 - 记录管理（RM = Record Manager）
├── index/          # 索引层 - 索引管理（IX = IndeX）
├── system/         # 系统层 - 系统管理（SM = System Manager）
├── transaction/    # 事务层 - 事务管理、并发控制
├── execution/      # 执行层 - 查询执行器（各类算子）
├── optimizer/      # 优化器 - 查询优化与计划生成
├── parser/         # 解析器 - SQL 解析
├── analyze/        # 分析器 - 语义分析
├── replacer/       # 替换器 - LRU 页面替换算法
├── recovery/       # 恢复层 - 检查点与日志恢复
├── common/         # 公共模块 - 配置
└── tests/          # 单元测试
```

- 语言：C++17
- 构建：CMake 3.16+
- 编译器：GCC 7.1+
- 依赖：flex, bison, readline
