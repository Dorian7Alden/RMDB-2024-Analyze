# 文档维护

维护 `my-analysis/` 下的教程文档。只检查最近查看和直接关联的文档，不全量扫描。

## 工作流

### 第 1 步：确定维护范围

- 用户指定了文件 → 只检查那些文件
- 用户未指定 → 检查当前打开的文件 + 其导航链接指向的相邻文档
- 不扫描整个 `my-analysis/` 目录

### 第 2 步：逐文件检查

对每个范围内的文件，依次检查以下项目：

**文件名与内容匹配：**
- 文件名中的话题前缀是否反映文档实际覆盖范围？
- 如果文档内容已经超出文件名范围很多 → 建议拆分或改名

**交叉引用有效：**
- 文档内引用的文件路径是否存在？
- 导航链接（上一节/下一节）指向的文件是否存在？
- 引用的源码路径和行号是否仍然准确？

**内容归属合理：**
- 有没有内容块更适合放在同目录的其他文档中？
- 有没有某个主题展开过多、应该独立成文的？
- 判断方式：通过文件名判断各文档的职责范围，不需要精读每个文档

**格式一致性：**
- 是否遵循一句一段？
- 是否所有解释性段落有指示性标签开头？
- 首次出现的缩写是否给了全称？
- 末尾是否有导航链接？

### 第 3 步：修复并提交

- 能直接修的问题直接修（比如更新路径、补充缺失的导航链接）
- 需要用户决策的提出来（比如"这个文档要不要拆成两个？"）
- 修复完成后激活 `commit` skill 提交

### 第 4 步：汇报

简要告诉用户：
- 检查了几个文件
- 修复了哪些问题
- 有什么需要用户决策的事项

## 原则

- **聚焦最近查看**：不全量扫描，只看用户正在用的文档
- **一次性审核完毕**：省略中间过程，直接给结果
- **文件名即职责**：通过文件名判断内容归属，减少 token 消耗
