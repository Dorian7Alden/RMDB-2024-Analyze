# DBMS 教学流程

用户要求学习某个 DBMS 概念时，按以下 8 步工作流执行。每步依赖前一步的结果，不要跳步。

## 工作流

### 第 1 步：建立上下文

讲解任何内容前先给读者一个定位：

- 这个概念属于哪个层/模块？（storage / record / index / system / transaction / execution）
- 它是什么？（一句话定义）
- 为什么存在？（解决什么问题？）
- 在架构中处于什么位置？（关联到 `src/` 目录树中的位置）

保持简短——目的是给读者一个心理地图，不是写教科书。

### 第 2 步：联系已学层

让知识和已有认知挂钩，不要孤立地讲新概念：

- **相似处**："和[已学层]的 X 一样，这个 Y 也……"（比如 `.idx` 和 `.db` 都是定长页面文件、都按 Page 加载、都有文件头和页头）
- **差异处**："但和[已学层]不同，这里 Z 不一样，因为……"（比如 Bitmap vs Keys 数组、空闲链表 vs B+ 树结构）
- 如果对比维度超过 3 个，用对比表格呈现

目的是让读者感觉知识是连贯的，每层都在已有基础上构建。

### 第 3 步：源码优先

先展示代码，再解释含义。源码是第一手材料，讲解是辅助。

- 先打开头文件，展示类/结构体声明
- 给出文件路径和行号范围
- 让读者先看到代码长什么样
- 然后再解释代码的含义

格式：
```
**源码**：`src/xxx/xxx.h:10-50`

[代码块]
```

### 第 4 步：三层递进讲解

这是教学的核心结构。禁止一上来就讲具体方法或代码片段——必须从宏观到微观，由浅入深分三层。

**第一层：概念定义（What）**
先讲清楚它是什么、为什么存在、解决什么问题。给大图景——用 mermaid 流程图或文字概述。读者读完应该知道：输入什么、输出什么、分几个阶段、各阶段什么关系。

**第二层：具体实现（How）**
把概览中的每个阶段拿出来逐一深入。每块也遵循同样的模式：这块做什么 → 展示代码 → 解释逻辑 → 给例子。每次一块，按顺序来。读者读完应该知道：代码怎么写的、关键数据结构是什么、算法步骤如何。

**第三层：性能讨论（How well）**
全部讲完后补充性能视角：

- **为什么这样设计**：当前实现做了哪些取舍来提升性能？（比如为什么用哈希表而不是排序、为什么 reduce lock contention、为什么 bulk memcpy 而不是逐条插入）
- **瓶颈在哪**：哪些操作的性能关键路径是什么？锁争用、磁盘 I/O、内存拷贝、CPU 缓存？
- **优化空间**：当前实现的不足在哪？源码中有哪些注释标记了优化方向（TODO、注释掉的替代方案）？
- **与其他设计的对比**：如果改用另一种常见做法会怎样？（比如 lock coupling vs latch crabbing、no-wait vs wait-die、LRU vs Clock）

不与第一层混淆——第一层讲"是什么"，这一层讲"为什么是这个而不是别的"。

### 第 5 步：框架对比

对比 `db2026-x/`（比赛框架——学习者要写的）和 `src/`（参考实现——怎么写好的）：

- 每个差异都要解释参考实现**为什么**做这个选择，不止列变化
- 用对比表格：| 方面 | 框架 (db2026-x) | 参考 (src) | 为什么不同 |
- 框架已完整实现的内容简要提及即可——重点放在 TODO / 方法体为空的待实现部分
- 目标：学习者理解设计思路，不是抄代码

### 第 6 步：具体实例走读

抽象讲解不够，必须端到端走一个具体例子：

- 用具体的表（比如"student 表有 id INT, name STRING, age INT"）
- 用具体的值（比如"把 age=20 插入到节点 (18, 22, 25) 中"）
- 逐步追踪数据流："当 X 调用 Y 时，实际发生的是……"
- 展示操作前后的状态变化

### 第 7 步：范围评估

讲解完后，评估这个主题需要哪些文档：

- 这层有交互文档了吗？（每层必须有一个，检查 `my-analysis/0X-.../` 目录）
- 这层有 API 速查文档了吗？（每层必须有一个）
- 这个主题需要数据结构总览图吗？（每层的第一个文档必须有）
- 是否有用户反复追问的概念，应该单独成文？

告诉用户哪些已经有了、哪些值得新建。

### 第 8 步：交接

问用户："要不要把刚才的内容写成文档？"

- 要 → 激活 `write` skill 来写文档
- 不要 → 到此为止，教学完成

讲解内容写成文档后，`maintain` skill 会审核是否遵循了教学原则（三层递进、框架对比、实例走读等），讲解时就要以能通过审核为标准。

## 核心原则

- **先是什么、再为什么、最后在哪里用**：定义概念 → 解释动机 → 展示使用场景
- **源码是最终目标**：学 DBMS 的最终目的是看懂源码、手写源码，每次讲解都必须追溯到实际代码
- **抽象必配实例**：每个抽象概念都要跟一个具体例子
- **尊重框架**：学习者的任务是把 `db2026-x/` 填满，帮他们理解设计思路而不是直接抄 `src/`
